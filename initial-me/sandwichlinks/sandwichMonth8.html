Huge Performance Upgrade on The New MCRLTR Application - Using A Tree Data Structure
I made a huge performance upgrade on a part of the MCRLTR application that was supposed to allow a user to select a Parent Category, and then based on that select the appropriate Child Category, and then based on then select a Grand-Child Category, and then a Great-Grand Child Category. This information being read is from a sharepoint which I had previously structured in a way for each element to know of its parent or children. Previously, the system was build in a way where a Parent Category is generated and then upon selection by a user a web service call to the Sharepoint list will be made in order to retrieve the Children.... This made this process incredibly slow for proper usage especially because the application will be expected to be used on mobile devices like the Ipad...
To improve this, the first idea was to build the entire list of categories locally into a Tree Structure (thanks to ONE web-service call), and then transverse through this list locally to get data and build Children Categories (this, in place of making web-service calls to sharepoint every time a new Category list needs to be built.)
